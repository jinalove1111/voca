## 1. 게임 시스템 (Priority 4) — 설계 + CTO 의견

- **별(⭐)**: 단일 보상 화폐 — 퀴즈/발음/미니게임/레벨업 미션 전부 이걸로 통일
- **스티커**: 수집형 보상(뽑기 + 확정 지급 두 경로), 희귀도(rarity) 시스템 있음
- **뱃지**: 별 100/300/500/1000개 달성, 연속학습 3/7/14/30일 — 스티커 컬렉션에 통합된 형태로 이미 존재
- **스트릭**: 연속 학습일 계산 이미 있음(`calcStreak`)
- **미션**: 오늘의 미션 4개(단어/예문/퀴즈/발음), 레벨업 미션(오답 복습 3회)
- **선물상자(Treasure Box)**: `GiftReveal.jsx` — 미션 4/4 완료 시 자동 오픈
- **다이어리 꾸미기**: 스티커를 배치하는 `DiaryPage.jsx` — Room Decoration의 미니 버전이 이미 있음
- **미니게임**: 풍선/낚시/피자/기차 (공유 로직 `MatchGameShell.jsx`)

### 요청하신 것 중 아직 없는 것과 내 의견

| 요청 | 의견 |
|---|---|
| **XP / Level** | 새 카운터를 또 만들기보다 **"레벨 = 누적 별 개수의 구간"**으로 파생시키는 걸 추천(예: 0~99별=Lv1, 100~299=Lv2...). 이미 있는 별 100/300/500/1000 뱃지 임계값과 자연스럽게 겹쳐서 그대로 "레벨업" 연출로 재활용 가능 — 새 스키마도, 새 화폐도 필요 없음. |
| **Coin / Gem** | **비추천** (현재 시점). 별 하나만으로도 충분히 게임화되어 있고, 화폐가 2~3개로 늘어나면 초등학생 입장에서 "이건 뭐에 쓰는 거지?"라는 혼란이 생김. 정말 필요하다면 "젬 = 유료/프리미엄 전용 화폐"처럼 **용도가 명확히 다를 때만** 하나 추가. |
| **Character / Pet** | CLAUDE.md 초기 설계엔 `PetCollection.jsx` 개념이 있었지만 현재는 스티커 시스템으로 대체된 상태. 완전히 새로 만들기보다, **스티커 중 "캐릭터" 카테고리를 만들어 다이어리에 배치된 캐릭터가 시간이 지나면 "친해지는"(작은 애니메이션 변화) 정도로 가볍게 확장**하는 걸 추천 — 새 시스템보다 기존 것의 깊이를 더하는 방향. |
| **Room Decoration** | 이미 `DiaryPage.jsx`(스티커 자유 배치)가 사실상 이 기능. "방"이라는 은유로 리브랜딩하고 배경(이미 `backgrounds.js`에 스트릭 기반 배경 잠금 시스템 있음)과 결합하면 완성도 있는 "내 방 꾸미기"가 별도 개발 없이 나옴. |
| **Daily/Weekly/Monthly Mission** | Daily는 이미 있음. Weekly/Monthly는 **스키마 확장 필요**(student_daily_progress를 주/월 단위로 집계하는 쿼리는 이미 가능 — v1.1의 관리자 대시보드가 이미 60일치를 가져오는 구조라 그대로 재사용 가능). 새 테이블 없이 기존 `student_daily_progress`를 주간/월간으로 집계만 하면 됨. |

### 추천 우선순위 (게임 시스템 안에서)
1. 레벨(별 파생) — 스키마 변경 없음, 오늘이라도 가능
2. Weekly Mission — 기존 daily 집계 로직 재사용, 스키마 변경 없음
3. Room Decoration 리브랜딩 — 기존 DiaryPage 확장
4. Pet/Character 카테고리 — 스티커 데이터에 필드 추가 정도(스키마 영향 적음)
5. Coin/Gem — 용도가 명확해지기 전까지 보류

## 2. 학습 시스템 (Priority 5) — 완성도

현재: 듣기(발음 재생) / 말하기(따라 말하기, blob.size 기반 판정) / 퀴즈(4지선다) / 쓰기(스펠링 테스트, 오늘 밤 추가) / 오답노트(퀴즈+스펠링 둘 다 있음) / 복습(레벨업 미션 + 스펠링 자동복습) — **요청하신 6개 중 5개 완성**.

남은 것: **Unit Test / Final Test**
- 설계 제안: 한 Unit의 모든 단어를 모아 "시험 모드"로 치르는 화면. 기존 QuizStep/SpellingQuestion 컴포넌트를 그대로 재사용(이미 단어 하나 단위로 만들어져 있어서 조합만 하면 됨), 결과를 `student_daily_progress`와는 별도로 "unit_test_results"에 기록하면 관리자가 "이 Unit을 통과했는지" 볼 수 있음 — **새 테이블 1개 필요** (스키마 변경, 사전 확인 필요).
- Final Test는 여러 Unit을 모은 것 — Unit Test 인프라가 있으면 자연 확장.

## 3. 학생 관리 구조 (Priority 6)

요청: XP/Level/Study Time/Progress/Accuracy/Wrong Answers/Streak 기록.

**이미 v1.1에서 구축된 것과 매핑:**
- Progress, Accuracy(퀴즈 정답률), Wrong Answers(많이 틀린 단어), Streak → `student_progress` + `student_daily_progress`에 이미 있음
- Study Time(학습 시간) → **아직 없음**. 화면에 머문 시간을 재는 건 배터리/성능 트레이드오프가 있어서, "오늘 미션 4개를 완료하는 데 걸린 시간" 정도의 가벼운 근사치를 추천(정확한 초 단위 트래킹보다 안정적).
- XP/Level → 위 1번 참고(별에서 파생, 새 필드 불필요)

**결론**: 학생 관리 구조는 사실 이미 90% 완성되어 있고, Study Time만 추가하면 요청사항을 거의 다 충족합니다.

## 5. AI 준비 구조 (Priority 8) — API 연결 없이 구조만

**원칙**: 지금은 AI API를 연결하지 않지만, 나중에 연결할 때 기존 코드를 다시 뜯어고치지 않도록 "AI가 들어갈 자리"를 인터페이스로 미리 비워두는 것.

이미 그런 패턴이 있음 — `speech.js`의 `transcribeViaServerSTT()`:
```js
// 이미 존재하는 실제 코드 패턴 (src/utils/speech.js)
export async function transcribeViaServerSTT(audioBlob) {
  if (!audioBlob) return null
  console.warn('[speech] transcribeViaServerSTT: not wired up yet...')
  return null  // 나중에 fetch('/api/transcribe', ...)로 교체만 하면 됨
}
```

**같은 패턴을 다른 AI 기능에도 적용 추천:**
- `AI Feedback`: `weeklyReport.js`의 `buildWeeklyReport()`가 이미 "규칙 기반이지만 AI가 하는 것처럼 보이는" 리포트를 만들고 있음 — 나중에 이 함수의 반환값을 `generateAIFeedback(data)`라는 이름의 async 함수로 감싸고, 내부를 규칙 기반 → AI 호출로 교체하면 호출부(AdminScreen.jsx)는 코드 수정 없이 그대로 작동.
- `AI Writing`(문장 검사): `isSpellingCorrect()`처럼 지금은 정확히 일치하는지만 보는 채점을, `checkSentence(text)`라는 별도 함수로 미리 분리해두면(지금은 규칙 기반: 길이/키워드 체크 등) 나중에 AI로 교체가 쉬움.
- `AI Speaking`: 이미 있는 "녹음 성공 = 연습 완료" 판정을(`blob.size > 0`) 그대로 두고, `gradesPronunciation(blob)` 같은 별도 함수로 감싸서 나중에 STT 연동 시 그 함수 내부만 교체.

**공통 원칙**: 지금 규칙 기반으로 동작하는 채점/피드백 로직을 "그 기능의 이름을 가진 별도 함수"로 분리해두면, 나중에 AI를 붙일 때 호출부(컴포넌트)는 건드릴 필요가 없음 — 오늘 이미 `spelling.js`/`weeklyReport.js`로 로직을 분리해온 방식과 정확히 같은 패턴.

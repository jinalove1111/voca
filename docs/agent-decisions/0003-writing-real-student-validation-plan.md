**관련 문서**: `0002-writing-feature-design-review.md`(설계·구현 기록), `handoff.md` 2026-07-20 (4차)(QA·배포 기록), 커밋 `21d78a1`/`bc17e5d`/`c752189`

## 배경

Writing(스펠링 복습 큐) MVP는 2026-07-20 전체 학생(111명)에게 배포 완료. 코드 빌드·회귀 테스트는 모두 PASS했지만, "실제로 학습에 도움이 되는가", "아이들이 억지로가 아니라 계속 쓰고 싶어하는가"는 코드 리뷰만으로 확인할 수 없다. 이 문서는 1주간의 실사용 관찰 계획이며, 이 단계에서는 앱 코드를 변경하지 않는다. 새 기능 추가도 하지 않는다.

## 1. 테스트 그룹 권장

이미 전체 학생에게 배포되어 있으므로 "그룹 분리 배포"가 아니라 "교사가 집중 관찰할 대상"을 정하는 것이 실질적 의미다. 권장 구성:

- 레벨이 다른 반 2개 이상, 총 10~15명 규모 (전수 관찰은 현실적으로 불가능).
- 그중 평소 스펠링 오답이 잦은 학생 3~5명을 의도적으로 포함 — 복습 큐가 가장 빨리 쌓일 가능성이 높은 대상이며, 아래 실패 기준(큐 무한 누적)이 가장 먼저 드러날 그룹.
- 모바일 위주로 앱을 쓰는 학생 최소 1~2명 포함 — 포커스/모바일 이슈 관찰용.

## 2. 관찰 체크리스트 (교사용)

**어디서 머뭇거리는가(hesitate)**
- 오답 1~2회까지는 아무 힌트/음성도 안 나옴(3회째부터 음성 잠금 해제) — 잠긴 음성 버튼을 이유 없이 계속 누르는지
- "복습 큐" 단어가 화면상 새 단어와 구분 없이 똑같이 보임 — "이거 전에 했던 거 아닌가?" 하는 반응이 있는지

**어디서 그만두는가(quit)**
- ⚠️ **최우선 관찰 항목**: 선물 팝업을 닫으면 예고 없이 강제 복습 화면으로 진입하고, 이 화면에는 나가기/뒤로가기 버튼이 없음(끝까지 풀어야 종료). 특히 큐에 여러 날 전 단어가 섞여 있을 때, 아이가 기기를 반납하거나 뒤로가기를 시도하는지 관찰
- 4번째 오답부터 정답이 공개되고 한 번 더 따라 써야 다음으로 넘어감 — 이 단계에서 포기하는 아이가 있는지

**반복되는 스펠링 오답 패턴**
- 같은 단어가 복습 큐에 계속 남아있는지, 아니면 며칠 내에 해결되는지(교사 체감 기준)
- 정답 공개 후 "그냥 베껴 쓰기"로 통과하는 경우가 많은지 — 실제로 외운 게 아니라 화면의 정답을 옮겨 적고 넘어가는 패턴

**재시도가 부담스러운가(frustrating)**
- 며칠 전 틀린 단어를 다시 틀렸을 때, 새 단어와 똑같은 3회 무힌트 절차를 다시 거침 — "또 틀렸네"라는 반응이 있는지
- 한 세션 안에서 같은 단어를 두 번 마주치는 경우(수업 중 1회 + 세션 끝 강제 복습 1회)가 있는지, 그때 아이 반응

**진행 상황을 이해하는가**
- 정답을 맞혔을 때 뜨는 "🎉 예전에 틀렸던 단어를 완전히 익혔어요!" 배지를 아이가 실제로 읽고 의미를 이해하는지, 아니면 그냥 지나치는지
- "왜 특별한 표시가 떴지?"라고 묻는 아이가 있는지(문제를 볼 때는 새 단어와 복습 단어가 시각적으로 구분되지 않음)

**숙제 후 자발적으로 계속하는가**
- 강제 복습 화면 진입 후 바로 앱을 끄는 아이가 있는지(선물 받은 직후 "끝났다"고 느꼈다가 복습을 강요받는 흐름)
- 반대로 🎉 배지 등장 이후 오히려 더 하고 싶어하는 긍정적 반응이 있는지도 함께 기록

**모바일/포커스 문제**
- 빈 칸으로 제출했을 때 아이폰/아이패드 사파리에서 화면상 아무 반응이 없어 보이는지(포커스 기능이 iOS Safari에서 100% 보장되지 않음 — 코드에도 명시된 한계) → 이 경우 "고장났다"고 느끼는지 관찰
- 문제 전환마다 키보드가 깜빡이며 사라졌다 나타나는 현상이 있는지, 그로 인해 오타/오터치가 느는지
- 음성 잠금 해제 버튼이 입력창 바로 위에 있어 모바일에서 오터치로 눌리는 경우가 있는지

## 3. 기존 데이터로 이미 측정 가능한 최소 지표 (신규 로깅 없음)

새 계측 인프라를 만들지 않고, 지금 있는 데이터로 확인 가능한 것만 사용한다:

- **스펠링 정답률 추이**: `history[date].spellingCorrect` / `.spellingTotal` (일자별 첫 시도 기준, `useStudent.js`) — 주간 추이를 볼 수 있으나 현재 관리자 화면에 노출되지 않으므로 이번 검증 기간에는 원본 데이터를 직접 조회해 확인.
- **의무 학습 대비 자발적 지속 사용**: `history[date].giftsToday`(하루 4/4 완주 횟수) vs `categoriesCompleted`(숙제 완료 최소 기준 4) — 기존에 이미 "의무 vs 자발적 지속"을 구분하는 유일한 신호.
- **복습 큐 적체/해소 추이**: `spellingReviewQueue`(progress_data jsonb 내부 배열)는 현재 스냅샷만 저장되고 시계열 이력은 없다. 새 로깅을 추가하지 않는 범위 내에서, 검증 1주 동안 매일 1회 읽기 전용 조회로 관찰 대상 학생들의 큐 길이를 수동 기록하는 절차를 권장(코드 변경 아님, 기존 데이터를 매일 스냅샷 뜨는 운영 절차).
- **명시적으로 지금은 측정 불가능한 것** (이번 검증 지표로 제안하지 않음): 세션 이탈 이유, 문항별 재시도 횟수(로컬 상태로만 존재, 저장 안 됨), 머뭇거림/세션 길이(타임스탬프 없음), 정답 공개 후 베껴쓰기인지 실제 암기인지의 구분(체크리스트의 정성 관찰에 의존).

## 4. 성공 기준

- 관찰 대상 학생 대부분에서 주간 동안 복습 큐에 들어온 단어의 상당수가 그 주 안에 정답으로 해소됨(큐가 계속 쌓이기만 하지 않음).
- `spellingCorrect`/`spellingTotal` 정답률이 기능 도입 전 대비 하락하지 않음.
- `giftsToday`(자발적 4/4 완주) 빈도가 기능 도입 전 대비 뚜렷이 감소하지 않음.
- 교사 체크리스트 상 강제 복습 화면에서의 반복적 이탈/짜증 신호가 소수(예외적 사례) 수준에 그침.

## 5. 실패 기준 (경고 신호 — 이 중 하나라도 뚜렷하면 롤백/보완 검토)

- 관찰 대상 다수에서 `spellingReviewQueue` 길이가 주간 내내 줄지 않고 계속 쌓임(구조적으로 큐에 크기 제한이 없어 이 위험이 코드상 존재).
- 선물 팝업 직후 강제 복습 화면 진입 시점에 반복적으로 앱을 끄거나 뒤로가기를 시도하는 패턴이 다수 관찰됨(이 화면은 현재 나가기 버튼이 없음).
- `giftsToday` 빈도가 기능 도입 전 대비 뚜렷이 감소 — 강제 복습이 자발적 지속 사용을 깎아먹고 있다는 신호.
- 교사가 "정답 공개 후 베껴쓰기로 통과하는 경우가 흔하다"고 다수 보고 — 배지는 뜨지만 실제 암기가 아닌 패턴.
- 모바일에서 빈 칸 제출 시 반응 없음으로 인해 아이들이 "고장났다"고 느끼는 사례가 다수 관찰됨.

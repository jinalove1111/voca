# 콘텐츠 파이프라인 — 교과서 임포트 워크플로우 설계
## "붙여넣기 → 자동 처리 → 사용 준비 완료"

- **목적**: 원장이 교과서 본문(영어+번역)을 붙여넣기만 하면 단어/문장/
  지문이 자동으로 처리되어 학생이 바로 학습할 수 있는 상태까지 이르는
  **워크플로우**(UX 단계 + 검수 정책 + 저장 전략 + 실패 처리 + 무중단
  도입 순서)를 설계한다. **이 문서는 워크플로우 설계이며, JSON 교환
  스펙/필드 매핑/DB 상세/멱등 임포트 전략은 이미
  `docs/reading/01-textbook-import-spec.md`(이하 "01번 스펙")가 버전 1.0으로
  정의해 두었다 — 여기서는 그 스펙을 그대로 목표 산출물(target format)로
  채택하고 재정의하지 않는다.** 파싱/문장분리/단어추출/중복매칭의 상세
  규칙은 `docs/research/lesson-analyzer-design.md`(별도 세션 동시 작성 —
  이 문서는 참조 포인터로만 언급, 규칙 중복 정의 없음)의 몫이다.
- **범위 확인**: 코드/SQL/설정 파일 변경 없음. 순수 설계 문서.
- **작성일**: 2026-07-23

## 0. 한 문장 요약

새 테이블이나 새 API 엔드포인트를 만들지 않고, 기존 관리자 화면에
"③ 붙여넣기" 입력 방식을 하나 더 추가해 붙여넣은 텍스트를
01번 스펙의 JSON 구조로 변환한 뒤, 01번 스펙이 이미 정의한
자연키-upsert 멱등 임포터로 `textbooks/units/words/passages/
passage_sentences/sentence_words` 6개 기존 테이블에 저장한다.

## 1. 왜 새 설계가 아니라 "접합"인가 (선행 조사 결론)

작업 지시에 따라 아래를 먼저 확인했다.

| 확인 대상 | 실측 결과 | 이 설계에 미치는 영향 |
|---|---|---|
| `src/components/AdminScreen.jsx` 단어 업로드 | 이미 3가지 입력 방식 공존: 엑셀(`.xlsx/.csv`), PDF 텍스트 추출 후 파싱, 그리고 `PassageEditor.jsx`의 지문 문장 "붙여넣기→미리보기→저장"(`pasteText`/`showPaste` state, `splitPassageText`) | "붙여넣기 → 파싱 미리보기 → 확인 후 저장"은 **이미 이 코드베이스의 확립된 UX 패턴**이다. 새 패턴을 발명하지 않고 그대로 확장한다(§2) |
| `supabase_v3_1_textbooks.sql` | `textbooks`/`class_textbooks` 이미 존재, 반→교재→유닛→단어 계층 확정. "중복 설계 금지" 지시대로 이 위에 얹는다 | 교재 엔티티를 새로 설계할 필요 없음 |
| `supabase_v3_3_reading.sql` / `v3_4` | `passages`/`passage_sentences`(+학습 메타 4컬럼)/`sentence_progress`/`sentence_words` 이미 존재, 전부 `unit_id` 상속으로 반→교재→유닛 계층 자동 상속 | 지문/문장 저장소도 이미 완비 |
| `docs/reading/01-textbook-import-spec.md` | **교재 일괄 임포트 JSON 스펙이 버전 1.0으로 이미 정의돼 있다** — 최상위 JSON 구조, 필드→DB 매핑 표, 검증 규칙(§5), 자연키 upsert 멱등 전략(§6, 삭제 없음/드라이런 필수/DDL 없음/폴백 안전)까지 전부 명시. 단, "임포트 도구 구현은 이 문서의 범위가 아니다(향후 구현용 — 지금 구현 금지)"라고 스스로 선을 그어 놓음 | **이 발견이 이 설계 문서의 핵심 전제다.** 신규 테이블/DDL이 사실상 불필요하고(§4), 이 문서가 할 일은 "01번 스펙의 JSON을 어떻게 만들고(§2 입력), 어떻게 검수하고(§3), 언제 실행할지(§6)"를 설계하는 것으로 좁혀진다. 01번 스펙의 필드 매핑/검증 규칙/멱등 전략을 여기서 재작성하지 않는다 — 필요할 때마다 "01번 스펙 §n 참고"로 인용한다 |
| `ROADMAP.md` 최상단 | v3.3 코드 완료(SQL 실행 대기), v3.1 활성화 완료. api 함수 **12/12로 Vercel Hobby 한도 도달(여유 0)** | §5 저장 단계와 §7 무중단 도입 순서에 신규 `api/*.js` 파일을 추가하지 않는다는 제약으로 직접 반영 |
| `docs/reading/02~08` | 문장분리/청킹/빈칸/문법분류/중요도기준/격려문구/데일리미션 UX가 별도 스펙 문서로 이미 존재 | 이 문서에서 상세 규칙 재정의 안 함 — 필요 시 각 문서로 포인터만 |

## 2. 입력 단계 — 붙여넣기 UX

### 2.1 어디에 놓는가

새 화면을 만들지 않는다. `AdminScreen.jsx`의 반 관리 탭 안에 이미
"엑셀 업로드 / PDF 업로드"가 나란히 있는 것과 같은 자리에 **세 번째
탭 "③ 교과서 붙여넣기"**를 추가한다. `PassageEditor.jsx`가 지문
문장에 이미 쓰고 있는 `pasteText`/`showPaste` 패턴을 컴포넌트
레벨로 승격한 것 — 새 상호작용 방식이 아니라 기존 것의 재사용이다.

### 2.2 입력 형태

원장이 붙여넣는 것은 교재 원문 그대로다. 정해진 구분자 문법을
강요하지 않는다(원장에게 마크다운/JSON을 배우게 하지 않는다는 원칙).
대신 아래 3개 텍스트 영역으로 입력을 나눠, "무엇을 어디에 붙여넣어야
하는지"만 명확히 한다 — 실제 파싱 규칙은 `lesson-analyzer-design.md`
소관이므로 여기서는 입력란 구성만 정의한다.

1. **교재/유닛 식별** — 드롭다운(기존 반 선택 + 그 반의 교재
   선택, 없으면 "새 교재"/"새 유닛" 인라인 생성). 텍스트 파싱 대상이
   아니라 폼 입력이다.
2. **단어 목록 붙여넣기(선택)** — 엑셀 업로드가 이미 처리하는 것과
   동일한 자유형 붙여넣기(줄바꿈/탭/쉼표 구분). 이미 있는 엑셀 파서
   로직(`AdminScreen.jsx`)을 텍스트 입력에도 그대로 재사용 —
   "파일 선택" 대신 "텍스트 붙여넣기"라는 입력 경로 하나가 늘 뿐,
   파싱 함수는 공유한다.
3. **지문 본문 붙여넣기(영어, 필수 — 지문을 만들 경우)** +
   **번역 붙여넣기(한국어, 선택)** — `PassageEditor.jsx`의
   `splitPassageText`가 이미 하는 일과 동일한 자리. 영어/한국어를
   별도 텍스트 영역으로 나누는 것은 `02-sentence-splitting-rules.md`가
   정의할 "줄 수 불일치 시 처리"를 UX에서 미리 줄이기 위함(문장 수가
   다르면 자동 정렬이 아니라 나란히 두 컬럼으로 보여주고 사람이
   맞춰보게 한다 — 상세 정렬 규칙은 lesson-analyzer 몫).

### 2.3 붙여넣는 즉시 하는 것 — 입력 검증(파싱 이전)

파싱/분석 전에 **입력 자체의 형식 문제**를 즉시 알려준다(이 레이어는
분석 규칙이 아니라 방어적 UX이므로 여기서 정의):

- 빈 입력, 공백만 있는 입력 → 저장 버튼 비활성화(엑셀 업로드가
  `preview.length === 0`일 때 이미 하는 것과 동일 패턴).
- 지문 본문에 영어 문자가 사실상 없음(예: 붙여넣기 실수로 한국어
  본문을 영어 칸에 넣은 경우) → 경고 배너, 저장은 막지 않되 확인
  요구(엑셀 업로드가 "기존 단어 덮어쓰기" 시 `window.confirm`하는
  것과 같은 급의 확인).
- 붙여넣은 텍스트가 비정상적으로 큼(예: 교재 전체를 한 유닛에
  붙여넣은 경우, 대략 지문 20개/문장 200개 초과 등 임계치는 구현
  시점에 실측 조정) → "한 유닛에 비해 많아 보여요, 유닛을 나눠
  붙여넣을까요?" 안내. 차단은 아님 — 원장 판단 존중.

### 2.4 미리보기

기존 엑셀 업로드 미리보기 테이블(유닛/단어/뜻 3열, 상위 20행)과
동일한 시각 언어로, 아래 두 블록을 나란히 보여준다.

- **단어 미리보기**: 엑셀 업로드와 동일 테이블.
- **문장 미리보기**: `PassageEditor.jsx`가 지문 편집 중 이미 쓰는
  문장 리스트 UI 재사용(영어/한국어 2열 + §3에서 정의할 검수 배지).

이 미리보기 단계에서는 아직 아무것도 저장하지 않는다 — 저장은
§3 검수를 통과한 뒤 §5의 임포터가 실행한다.

## 3. 검수 단계 — 사람 확인 vs 자동 확정

원칙: **"이미 검증된 규칙이 있는 것은 자동 확정, 원장 고유 판단이
필요한 것은 반드시 확인."** 아래 표는 무엇이 자동/수동인지 정책만
정하며, 판정 로직 자체(문장 분리 정확도, 단어 추출 정확도, 중복 매칭
알고리즘)는 `lesson-analyzer-design.md`가 정의한다.

| 항목 | 자동 확정 | 사람 확인 필수 | 근거 |
|---|---|---|---|
| 단어-기존 단어 중복 매칭 (같은 유닛 내 동일 단어) | 완전 일치(대소문자 무시)는 자동 스킵 | 유사(오탈자 의심) 매칭은 배지만 표시, 확정은 원장 클릭 | 01번 스펙 §6은 `(unit_id, normalizeAnswer(word))`을 자연키로 정의 — 완전 일치는 이미 "재사용" 규칙이 확정돼 있으므로 자동 처리해도 스펙 위반이 아님. 유사 매칭 판정은 lesson-analyzer 몫이라 이 문서는 "표시만 하고 자동 확정은 안 함"이라는 정책만 못박는다 |
| 문장 분리 경계 | 명확한 마침표/느낌표/물음표 종결은 자동 확정 | 축약형(Mr./예문 인용부호 등) 경계는 배지로 표시, 원장이 병합/분리 가능 | `02-sentence-splitting-rules.md` 소관 — 이 문서는 "모호한 경계는 항상 사람이 최종 확인"이라는 UX 계약만 규정 |
| 영어-한국어 문장 대응(줄 수 일치 시) | 줄 수가 정확히 같으면 순서대로 1:1 자동 대응 | 줄 수가 다르면 자동 대응 금지 — 나란히 보여주고 원장이 드래그/버튼으로 짝짓기 | 오대응된 번역이 그대로 학생 화면에 노출되는 리스크가 "빠른 처리"보다 크다(원장·학부모 신뢰가 아이보다 먼저 깎인다는 제품 리뷰 문서 10의 우선순위와 동일 정신) |
| `is_key_sentence`/`importance_level`/`grammar_point`/`chunks` (v3.4 학습 메타) | 없음 — 전부 기본값(false/1/null/null)으로 저장 | 전부 사람 입력 전용. 01번 스펙 §5.3과 `chunks` 스키마 주석("수동 입력 전용 — AI 자동 분할 없음")이 이미 못박은 규칙 | 규칙 재정의 아님 — 그대로 인용. 붙여넣기 임포트 단계에서는 이 4개 필드를 **아예 건드리지 않는다**(기존 지문 편집 화면에서 나중에 채움) |
| `linked_words`(문장-단어 연결) | 없음 | 이번 임포트 플로우 범위 밖 | 01번 스펙 §5.4/`sentence_words` SQL 주석("이번 Phase에는 연결 UI 없음")과 동일 — 이 파이프라인도 안 다룸 |
| 반/교재/유닛 대상 지정 | 기존 반+기존 유닛 선택 시 자동(재확인 불필요, 이미 `select` UI로 명시적 선택) | "새 교재"/"새 유닛" 생성은 이름 확인 1회(오타로 유사 유닛이 중복 생성되는 사고 방지) | 엑셀 업로드가 지금도 "반은 항상 선택한 반"이라고 명시하는 것과 같은 안전장치 |
| 기존 데이터 덮어쓰기 | 없음 | 대상 유닛에 이미 단어/지문이 있으면 항상 확인 다이얼로그 | `setClassWords`가 "delete-then-insert"로 유닛의 기존 단어를 전량 교체하는 실제 동작(`src/utils/wordLibrary.js:538` 부근 주석)이 이미 있고, 엑셀 업로드는 이를 `window.confirm`으로 막는다 — 붙여넣기 경로도 동일 안전장치 필수. 지문은 01번 스펙 §6 표에서 "삭제 없음(update만)"이라 상대적으로 안전하지만, 단어 축은 여전히 파괴적이므로 확인 유지 |

검수 화면에서 원장이 보는 것은 **하나의 통합 미리보기**(단어 표 +
문장 표, 각 행에 "자동 확정"/"확인 필요" 배지)이지 여러 단계로
쪼개진 마법사가 아니다 — "확인 필요" 배지가 하나도 없으면 저장
버튼 클릭 한 번으로 끝나고(교재가 이미 깔끔한 경우 대다수), 배지가
있으면 그 행만 펼쳐서 고치면 된다.

## 4. 저장 단계 — 기존 테이블에 어떻게 들어가는가

### 4.1 신규 테이블/컬럼이 필요한가 — 결론: 원칙적으로 불필요

§1에서 확인했듯 `textbooks`(v3.1), `class_textbooks`(v3.1),
`units.textbook_id`(v3.1), `passages`/`passage_sentences`(v3.3),
`passage_sentences`의 학습 메타 4컬럼(v3.4), `sentence_words`(v3.4)가
모두 이미 존재하고, 01번 스펙이 이 6개 테이블로의 필드 매핑을
버전 1.0으로 완결했다. 이 파이프라인은 **그 매핑을 그대로 쓴다**.
새 DDL을 이 문서에서 설계하지 않는다(설계할 필요가 없다는 것 자체가
결론).

### 4.2 유일하게 검토할 만한 추가 — "임포트 실행 로그" (선택, 필수 아님)

붙여넣기 임포트가 반복 사용되면 "언제 누가 무엇을 얼마나
넣었는지"를 사후 확인할 필요가 생길 수 있다. 다만:

- 이는 **콘텐츠 데이터가 아니라 운영 로그**이므로 없어도 학생/원장
  기능이 전혀 깨지지 않는다(추가하지 않아도 §7의 무중단 도입에
  영향 없음).
- 이미 `product_events`(`supabase_v3_2_product_events.sql`, 완전
  익명·비파괴 이벤트 로그)라는 정확히 이 용도의 테이블이 있다.
  새 테이블을 또 만들지 않고, 임포트 실행 시
  `product_events`에 이벤트 타입 하나(예: `event_type =
  'admin_textbook_import'`, `meta`에 교재명/유닛명/단어수/문장수/
  자동확정수/수동확정수를 jsonb로) 추가하는 것으로 충분하다 —
  **신규 테이블 0개, 기존 스키마에 이벤트 타입 값 하나 추가**는
  DDL 변경조차 아니다(컬럼이 이미 자유 타입 `event_type
  text`/`meta jsonb`라면). 실측은 구현 시점에
  `supabase_v3_2_product_events.sql` 재확인 필요 — 이 문서는
  가능성만 기록하고 단정하지 않는다.
- 이 로그가 없어도 파이프라인 자체는 완전히 동작한다 — "있으면
  좋고, 없어도 무방한" 부가 기능으로 명시해 둔다(과설계 방지).

### 4.3 저장 순서(01번 스펙 §6 인용, 재정의 아님)

붙여넣기로 만든 JSON(§2~3을 거쳐 확정된 것)을 01번 스펙 §6의
자연키 upsert 순서 그대로 적용한다: `textbooks.name` →
`(textbook_id, units.name)` → `(unit_id, normalizeAnswer(word))` →
`(unit_id, title)`(passages) → `(passage_id, position)`
(sentences) → `(sentence_id, word_id)`(sentence_words, 이번
파이프라인 범위 밖이라 사실상 항상 빈 배열). 삭제 없음 원칙도
그대로 상속 — 단, §3에서 정한 대로 **단어만은** 예외적으로
`setClassWords`의 기존 delete-then-insert 동작을 그대로 쓴다(01번
스펙의 words 자연키 upsert와 실제 코드 동작이 다르다는 점을 아래
4.4에서 정직하게 기록).

### 4.4 정직한 불일치 기록 — words 테이블 upsert vs 기존 코드 동작

01번 스펙 §6 표는 word의 자연키를 `(unit_id,
normalizeAnswer(word))`로 "있으면 재사용, 없으면 insert"라고
정의한다. 그런데 실제 코드(`wordLibrary.js` `setClassWords`,
§1에서 확인)는 유닛 저장 시 **기존 단어를 전량 delete 후
재insert**하는 다른 전략을 쓴다(단, 오디오/예문은 단어 텍스트로
매칭해 이월). 이 파이프라인을 구현할 때 두 가지 선택지가 있다.

1. **기존 `setClassWords` 그대로 재사용**(delete-then-insert) —
   구현 비용 최소, 기존 엑셀/PDF 업로드와 완전히 같은 동작이라
   회귀 위험 없음. 단, 01번 스펙의 "삭제 없음" 원칙과 문자 그대로는
   다르다.
2. **01번 스펙의 자연키 upsert를 words에도 새로 구현** — 스펙과
   완전히 일치하지만, 기존 검증된 `setClassWords` 로직을 건드리지
   않는다는 안정성 원칙(저장소 헌법 규칙 1: 기존 기능을 절대 깨뜨리지
   않는다)에는 오히려 반한다.

**이 문서의 권고: 1번(기존 `setClassWords` 재사용)을 채택**하고,
01번 스펙에는 "붙여넣기 파이프라인은 words 축에 한해 기존
delete-then-insert 전략을 따른다"는 각주를 향후 그 문서 작업자가
append하는 것을 제안한다(이 문서 자신은 01번 스펙을 수정하지
않는다 — append-only 원칙과 파일 소유권 원칙상 그 문서의 소유
세션이 반영). 근거: 지문/문장(§4.3의 나머지 축)은 이미 "삭제 없음"
전략의 신규 테이블이라 스펙 그대로 가도 안전하지만, `words`는
`word_status`/`spelling_review_queue`/`sentence_words` 등 여러
테이블이 `word_id`를 FK로 참조하는 오래된 핵심 테이블이라 새로운
upsert 전략을 지금 처음 도입하는 리스크가, 스펙 문구와의 사소한
불일치보다 크다.

## 5. Vercel 서버리스 함수 제약 — 신규 API 파일 추가 불가

`ROADMAP.md` 최상단 기준 `api/*.js`가 **12/12로 Vercel Hobby 한도에
이미 도달**(여유 0, 신규 파일 추가 시 배포 실패). 이 파이프라인은
사실 서버리스 함수가 거의 필요 없다 — §4에서 확인했듯 저장은 전부
anon key로 직접 쓰기 가능한 테이블들(`"allow anon all"` RLS 관례)
이므로 클라이언트가 Supabase JS SDK로 직접 upsert하면 된다(기존
`setClassWords`/`readingApi.js`의 저장 함수들이 이미 이렇게
동작). 파싱/검증 로직도 전부 순수 클라이언트 JS(`readingModel.js`
스타일의 import-0 순수 모듈)로 충분하다.

굳이 서버 쪽 개입이 필요한 지점은 단 하나 — **오디오 생성**
(`api/generate-audio.js`, 이미 존재하는 12개 중 하나)이다.
`setClassWords`가 새 단어 insert 후 `requestAudioGeneration`으로
이 기존 엔드포인트를 fire-and-forget 호출하는 것과 완전히 동일한
경로를 붙여넣기로 만든 단어에도 그대로 적용하면 된다 — **이 파이프
라인은 기존 12개 중 어느 것도 늘리지 않고, 있는 것을 그대로
재사용**한다.

만약 향후(이 문서 범위 밖) 이 파이프라인에 "붙여넣은 텍스트를
서버에서 재검증"처럼 서버 전용 로직이 필요해지는 순간이 오면,
새 파일을 만들지 말고 저장소가 이미 쓰고 있는 관례(예:
`api/admin-pin-actions.js`의 action-dispatch 패턴 — 여러
액션을 `{ action: '...' }` 바디로 분기해 파일 하나에서 처리)를
따라 **기존 액션 dispatch 파일에 새 action 케이스를 추가**하는
방식으로 흡수해야 한다. 이번 설계에서는 그 필요 자체가 없다는 것을
확인했으므로, 어느 기존 파일에 얹을지까지는 특정하지 않는다(실제
필요가 생겼을 때 그 시점의 구현 세션이 판단할 문제).

## 6. 실패/부분성공 처리

01번 스펙 §6이 이미 "검증 실패 시 아무것도 저장하지 않고 오류
목록 전체를 보고(all-or-nothing)"를 못박아 두었다(§5 인용). 이
파이프라인은 그 원칙을 UX로 구현하는 방식만 정의한다.

- **검증 단계 실패(저장 전)**: §3 미리보기 화면에서 오류가 있는
  행은 빨간 배지로 표시하고, 저장 버튼은 **오류가 1건이라도 있으면
  비활성화**(엑셀 업로드가 `!selectedClass`일 때 저장 버튼을
  `disabled`하는 것과 동일한 패턴 — 새 상태 관리 방식 도입 없음).
  경고(§2.3의 "확인 필요" 급)는 저장을 막지 않지만 오류(01번 스펙
  §5의 구조/필수필드 위반)는 막는다.
- **저장 중 실패(부분 쓰기 위험 구간)**: `textbooks/units` →
  `words` → `passages/passage_sentences` 순서로 여러 테이블에
  걸쳐 쓰기 때문에, 예를 들어 단어 저장은 성공했는데 지문 저장에서
  네트워크 오류가 나는 상황이 가능하다. 01번 스펙의 자연키 upsert
  전략(삭제 없음) 덕분에 **이 부분 실패는 데이터 파괴로 이어지지
  않는다** — 이미 저장된 단어는 그대로 남고, 실패한 지문만 다시
  시도하면 자연키가 겹치는 것들은 재사용되고 안 겹치는 것만
  새로 들어간다(진짜 트랜잭션이 아니라 "재시도해도 안전한 순서"로
  실패를 흡수하는 설계 — Postgres 멀티테이블 트랜잭션을 anon key
  클라이언트 SDK에서 직접 구현하지 않는 이 저장소의 기존 관례와
  일치, `setClassWords`도 트랜잭션 없이 delete→insert 두 단계임).
  실패 시 화면에는 "① 단어 32개 저장 완료 ② 지문 저장 실패(재시도
  가능) — 다시 시도해주세요"처럼 **어느 단계까지 성공했는지**를
  정직하게 보여준다(침묵 실패 금지).
- **v3.4 컬럼/테이블 부재(마이그레이션 미실행 환경)**: 이 파이프라인
  구현 시점에 `supabase_v3_3_reading.sql`/`v3_4`가 아직 미실행일
  가능성을 항상 고려해야 한다(§7에서 실제로 그 순서를 다룬다).
  `readingApi.js`의 `isMissingTableError`/`checkReadingTablesExist`/
  `checkSentenceColumnsExist` 패턴을 그대로 재사용해 테이블/컬럼
  부재를 감지하고, 지문 관련 입력란은 "SQL 실행 후 사용 가능"
  안내로 비활성화(기존 `PassageEditor.jsx`가 이미 하는 것과 동일) —
  단어만 붙여넣는 경로는 그 상태에서도 100% 정상 동작해야 한다
  (기존 테이블만 쓰므로 자동으로 충족됨).

## 7. 111명 실사용 서비스에 무중단으로 도입하는 순서

저장소 헌법 규칙 1(안정성 최우선)·규칙 9(마이그레이션 순서
무관 무중단)를 그대로 따른다. 이 파이프라인은 **새 테이블을
만들지 않으므로 신규 SQL 실행이 필수가 아니다** — 유일한 전제
조건은 이미 계획돼 있던 v3.3/v3.4 SQL의 실행 여부다.

1. **0단계(선행 조건, 이미 계획됨)**: `supabase_v3_3_reading.sql`,
   `supabase_v3_4_sentence_learning.sql`이 실행돼 있어야 지문/문장
   축이 동작한다(단어만 붙여넣는 경로는 이 없이도 동작). 이미
   ROADMAP에 "SQL 실행 대기"로 기록된 항목이라 이 파이프라인이
   새로 요구하는 게 아니다 — 기존 대기열에 편승한다.
2. **1단계(기능 플래그 뒤 배포)**: 이 저장소가 이미 쓰고 있는
   `config/features.js` Feature Flag 관례(`PROJECT_GUIDE.md` 문서
   지도의 "학원 운영 시스템 확장" 각주가 언급하는 실재 인프라)로
   `textbookPasteImport` 같은 플래그를 OFF로 배포한다. 학생 화면은
   전혀 건드리지 않으므로(규칙 12 — 학생 대상 신규 기능 금지 범위와
   무관, 이건 순수 관리자 도구) 플래그가 OFF여도 앱 전체가 아무
   영향 없다.
3. **2단계(관리자 소수 대상 사전 검증)**: 원장 본인 계정에서만
   플래그 ON으로 QA 반(유닛 0개, 실측 근거는 v3.1 SQL 주석의
   "QA 반 4개는 유닛 0" 기록)에 시험 붙여넣기 — **실제 111명 반
   데이터는 건드리지 않고** 워크플로우 전체(입력→검수→저장→오디오
   생성 트리거)를 검증.
4. **3단계(실 반 1개로 확대, 기존 데이터 있는 유닛)**: §3의
   "기존 데이터 덮어쓰기 확인 다이얼로그"가 실제로 뜨는지, 오디오
   재생성이 불필요하게 일어나지 않는지(`setClassWords`의 단어
   텍스트 매칭 이월 로직이 그대로 적용되는지)를 데이터가 이미 있는
   실 유닛 1개로 확인.
5. **4단계(전체 원장 노출)**: 플래그 ON 기본값 전환. 기존
   엑셀/PDF 업로드 탭은 제거하지 않고 그대로 둔다(붙여넣기가
   실패하거나 원장이 기존 방식을 선호할 때의 폴백 — 새 기능이
   기존 기능을 대체하는 게 아니라 나란히 추가되는 것, 규칙 1의
   정신).
6. **롤백**: 언제든 플래그를 OFF로 되돌리면 붙여넣기 탭만
   사라진다. 이미 저장된 단어/지문/문장은 기존 엑셀 업로드/
   `PassageEditor.jsx`로 계속 정상 조회·수정 가능(같은 테이블에
   같은 형태로 들어갔을 뿐이므로 "붙여넣기로 만들었다"는 흔적이
   데이터에 남지 않는다 — §4.2의 선택적 로그를 켜지 않는 한).

각 단계 전환 시 `npm run build` + 관련 `verify:*`(엑셀 업로드가
`wordLibrary.js`를 공유하므로 최소 `verify:persistence` 계열,
지문 축을 쓰면 `verify:reading`/`verify:sentence` 계열도) 통과를
전제로 한다(헌법 규칙 5) — 이 문서는 설계 문서이므로 실제 verify
스크립트 이름은 구현 세션이 `TESTING.md`에서 재확인해야 한다(추측
금지).

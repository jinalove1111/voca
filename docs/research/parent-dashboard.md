## 1. 현재 상태 요약 (지금 무엇이 있는가)

### 1.1 학부모 화면이 지금 보여주는 것

`ParentScreen.jsx` 기준: 오늘 학습 여부, 숙제 완료 여부, 누적 별,
연속 학습일수, 클리어 단어 수, (게임화 켠 반만) House 소속·티켓 잔액,
최근 7일 막대그래프(카테고리 완료 수), 퀴즈 정답률, 발음 연습 횟수,
취약 단어(최근 자주 틀린 단어 최대 5개), 그리고 `buildWeeklyReport()`가
조립하는 복사 가능한 텍스트 리포트(잘한 점 / 조금 더 노력하면 좋은 점).

### 1.2 `buildWeeklyReport()`가 이미 하고 있는 것 (선례로 존중해야 할 부분)

- **규칙 기반, AI 비용 0** — `CLAUDE.md` 규칙 7 그대로 실천 중. 이 문서의
  모든 신규 제안도 같은 원칙(규칙 기반 템플릿)을 따른다.
- **관리자 화면과 동일한 소스** — `fetchDashboardData`/
  `computeStudentStats`를 그대로 재사용해 새 Supabase 쿼리를 만들지 않음
  (`PAUL_PRINCIPLES.md` §7). 이 문서가 제안하는 항목도 우선 이 원칙을
  지킬 수 있는지("지금 데이터로 바로 가능")부터 판별했다.
- **압박이 아니라 성장 프레이밍** — 순위/랭킹 노출 안 함, "잘한 점"을
  항상 1개 이상 보장(`good.length === 0`이면 기본 문구로 폴백), "조금 더
  노력하면 좋은 점"도 없으면 칭찬 문구로 폴백.

### 1.3 조사에서 새로 확인한 사실 — `dailyRows`는 이미 최근 60일치를 들고 있다

`wordLibrary.js`의 `fetchDashboardData()`는 `student_daily_progress`를
**최근 60일** 윈도우로 조회한다(`cutoff.setDate(cutoff.getDate() - 60)`).
그런데 `ParentScreen.jsx`/`computeStudentStats()`는 이 중 `last7 =
r.dailyRows.slice(0, 7)`만 실제로 쓴다. 즉 **"패턴 형성" 계산에 필요한
최대 60일치 원자료가 이미 네트워크 요청 없이 메모리에 와 있다** — 아래
4축 설계에서 "지금 데이터로 바로 가능"으로 분류한 항목 다수가 새 쿼리가
아니라 이미 받아온 `r.dailyRows`를 더 길게 쓰는 것만으로 가능한 이유다.

### 1.4 학부모가 지금 못 보는 것 — 이 문서가 채우려는 공백

- 통계는 전부 **하루/누적 스냅샷**이고, "달라지고 있는가"(추세)를 말하는
  문장이 없다. 연속학습일수·클리어단어수 모두 "지금 값"이지 "지난주보다
  나아졌다/힘들어하고 있다" 같은 변화 판단이 아니다.
- 취약 단어는 "많이 틀린 단어 목록"만 보여줄 뿐, 학부모가 집에서 **무엇을
  하면 되는지**(예문으로 물어보기? 철자 불러주기? 그냥 다시 듣기?)로
  연결되지 않는다.
- 자신감(포기하지 않고 다시 시도하는지, 정답률이 널뛰는지 안정화되는지)을
  나타내는 신호가 화면에 아예 없다 — 정답률 숫자 하나뿐.

## 2. 설계 원칙 — 무엇을 배제하는가

아래는 **이번 설계에서 의도적으로 뺀** 지표와 이유(작업 지시의 "무의미한
통계 배제" 기준):

| 배제 지표 | 왜 배제하는가 |
|---|---|
| 총 학습 시간 누적 | 시간을 오래 썼다고 잘 배운 게 아니다 — 오히려 "느리다"는 오해를 낳을 위험. 행동 변화로 이어지는 다음 행동이 없다. |
| 누적 단어 수/누적 별 단순 나열 | 이미 화면에 있고(1.1), "많다/적다"만 알 뿐 무엇을 해야 할지 안 알려준다. 이 문서는 이 숫자를 없애자는 게 아니라 **추세/맥락 없이 새 지표로 추가하지 않는다**는 뜻. |
| 반 평균 대비 순위/백분위 | `PAUL_PRINCIPLES.md` §7이 이미 명시적으로 배제한 방향(비교가 아니라 성장) — 이 문서도 따른다. |
| 앱 접속 횟수/세션 수 | 실제 학습(카테고리 완료·퀴즈·발음)과 상관없는 "켰다 껐다" 신호라 행동 신호로 오독될 위험. |
| 정답/오답 원시 로그 나열 | 학부모가 소화하기엔 너무 저수준 — 반드시 "요약 문장 + 대표 사례 몇 개"로 가공해야 함(취약 단어 최대 5개로 이미 제한하는 기존 관례와 동일). |

**채택 기준**: 각 리포트 항목은 (a) 기존 테이블/컬럼에서 계산 가능하거나
계산 가능하도록 최소 수집을 추가할 수 있어야 하고, (b) 학부모가 읽었을 때
"그렇구나" 이상의 **구체적인 다음 행동**을 하나 이상 제시해야 하며, (c)
아이를 낙인찍지 않는 표현이어야 한다(순위/비교/"못한다" 단정 금지, 항상
현재진행형·격려형 문장).

## 3. 4축 설계

각 항목: **계산 근거**(테이블.컬럼) / **데이터 상태** / **문구 예시**
(실제 한국어 문안) / **학부모 다음 행동** / **우선순위**.

우선순위 3단계:
- 🟢 **지금 바로 가능** — 기존 테이블/컬럼만으로, 새 Supabase 쿼리도 거의
  불필요(1.3의 60일 윈도우 재활용 포함).
- 🟡 **수집 추가 필요** — 새 컬럼/테이블 또는 최소한 새 쿼리 윈도우 확장이
  필요하지만 기존 스키마 패턴(멱등 마이그레이션, GRANT 등 `CLAUDE.md`
  규칙 9/10) 안에서 무리 없이 추가 가능.
- 🔴 **장기** — 새로운 이벤트 로깅 체계나 세션 단위 타임스탬프 등 설계
  변경 규모가 크거나, 운영자 판단(예: 알림 채널 추가)이 선행돼야 함.

### 3.1 학습 습관 (Learning Habit) — "며칠 연속"이 아니라 "패턴이 자리잡고 있는가"

| 항목 | 계산 근거 | 상태 | 문구 예시 | 학부모 다음 행동 |
|---|---|---|---|---|
| **요일 패턴 형성** — 특정 요일에 학습이 몰리는지 vs 들쭉날쭉한지 | `student_daily_progress.date`(최근 60일, 1.3 참고) 요일별 분포. `categories_completed > 0`인 날의 요일 집계 후 표준편차/최빈 요일 계산(순수 함수, 규칙 기반) | 🟢 지금 바로 가능 | "화·목·토에 학습하는 패턴이 자리잡고 있어요." / "요일이 매번 달라지고 있어요 — 특정 요일을 정해보면 습관이 더 쉽게 자리잡아요." | 아이와 함께 "매주 이 요일엔 꼭 하기"를 1~2개 정해 캘린더에 표시 |
| **끊긴 후 회복 속도** — 하루 빠진 뒤 얼마 만에 다시 시작하는지 | `student_daily_progress.date`의 공백 구간 길이(gap) 계산 — 최근 60일 내 결측 구간과, 그 다음 첫 기록까지의 간격. 마지막 3~5개 공백의 평균 회복일수 | 🟢 지금 바로 가능 | "하루 쉬어도 보통 다음날 바로 다시 시작해요 — 좋은 회복력이에요." / "한 번 쉬면 며칠 이어서 쉬는 경우가 많아요." | 하루 빠졌을 때 그 다음날 "오늘 한 번만 하고 자자" 정도로 가볍게 다시 시작 유도 |
| **선호 학습 시간대** — 아침/오후/저녁 중 언제 주로 하는지 | 현재 `student_daily_progress`는 **날짜만** 있고 시각(시:분) 컬럼이 없음 | 🟡 수집 추가 필요 — `student_daily_progress`에 `first_activity_at timestamptz`(그날 첫 학습 시각) 컬럼 추가 제안. 멱등 `add column if not exists` + upsert 시 `coalesce(first_activity_at, now())`로 최초 1회만 기록(규칙 9) | "저녁 7~9시 사이에 주로 공부해요." | 그 시간대에 방해 요소(TV/게임)를 미리 정리해주기 |
| **숙제-시작 텀** — 숙제 배정일과 실제 학습일 사이 지연 | `daily_assignments.date` vs `student_daily_progress.date` 비교(같은 class_id, 근접 날짜) | 🟢 지금 바로 가능(단, 학부모 화면은 studentId만 갖고 class_id 조회가 추가로 필요 — 이미 `resolvedClassName`으로 보유) | "숙제 나온 날 바로바로 시작하고 있어요." | 숙제 공지 나오면 그날 알림 한 번 확인해주기 |

### 3.2 자신감 (Confidence) — 정답률 숫자가 아니라 "흔들림 vs 안정화" 신호

| 항목 | 계산 근거 | 상태 | 문구 예시 | 학부모 다음 행동 |
|---|---|---|---|---|
| **정답률 추세(2주 비교)** — 오르는 중인지, 떨어지는 중인지, 안정적인지 | `student_daily_progress.quiz_correct`/`quiz_total`(최근 60일). 최근 14일을 전/후 7일로 나눠 정답률 비교(단순 이동평균, AI 아님) | 🟢 지금 바로 가능 | "최근 2주간 퀴즈 정답률이 꾸준히 오르고 있어요(1주차 62% → 2주차 78%)." / "정답률이 오르내림이 커요 — 컨디션에 따라 차이가 있는 것 같아요." | 정답률이 낮은 날 패턴(요일/시간)이 보이면 그 시간을 피해서 시도해보기 |
| **같은 단어를 반복해서 틀리는지 vs 한 번 틀리고 넘어가는지** | `student_daily_progress.missed_word_ids`(jsonb, 날짜별) — 여러 날에 걸쳐 같은 word_id가 반복 등장하는지 집계 | 🟢 지금 바로 가능(이미 취약 단어 계산 로직이 하는 것과 같은 집계를 "반복 일수"로 재가공) | "대부분의 단어는 한 번 틀리고 나면 다음엔 맞혀요 — 실수를 잘 고쳐나가고 있어요." / "이 단어들은 여러 날에 걸쳐 계속 헷갈려해요 — 완전히 낯선 단어일 수 있어요." | 반복되는 단어만 골라 하루 3개씩 짧게 복습(전체 목록 말고 최대 3~5개로 제한 — 부담 방지) |
| **스스로 "모르겠어요"로 표시한 비율** — 자기 평가 기반 자신감 | `word_status.status`(known/unknown/skipped/mastered), `student_id`별 현재 스냅샷 비율 | 🟢 지금 바로 가능(현재 시점 스냅샷만) / 🟡 수집 추가 필요(시점 간 **추세**를 보려면 — 예: "2주 전보다 '모르겠어요' 비율이 줄었어요" — 현재 `word_status`는 최신 상태만 upsert되고 이력을 안 남김. 주기적 스냅샷 테이블 또는 `status_history` jsonb append 필요) | "스스로 '아는 단어'로 표시하는 비율이 늘고 있어요 — 자신감이 붙고 있다는 신호예요." | 아이가 '모르겠어요'로 체크한 단어를 같이 소리 내어 읽어주며 "이건 다음에 다시 보자" 정도로 가볍게 반응 |
| **입실시험 점수 변동폭** — 시험마다 점수가 안정적인지 | `entrance_test_results.score`/`total`/`submitted_at`(같은 학생의 여러 회차 비교) | 🟢 지금 바로 가능(단, 반이 입실시험을 실제로 쓰는 경우에 한함 — `entrance_tests`가 반별 opt-in) | "최근 시험 3번 모두 80점대를 유지하고 있어요 — 실력이 안정적이에요." | 시험 직전 컨디션(수면/급하게 옴 등)이 점수에 영향 없는지 관찰 |

### 3.3 약한 영역 (Weak Areas) — "무엇을 도와줄 수 있는지"까지 연결

| 항목 | 계산 근거 | 상태 | 문구 예시 | 학부모 다음 행동 |
|---|---|---|---|---|
| **자주 틀리는 단어 Top N + 유형** | `student_daily_progress.missed_word_ids` → `words` 테이블 조인해 단어 표시(이미 `ParentScreen.jsx`가 함). 유형 분류(철자 vs 발음 vs 뜻)는 어느 화면에서 틀렸는지(퀴즈 vs 쓰기시험 vs 발음)까지 구분하려면 `missed_word_ids`가 현재 "어느 활동에서 틀렸는지"를 구분 안 함 | 🟢 지금 바로 가능(단어 목록만) / 🟡 수집 추가 필요(활동 유형 구분 — `missed_word_ids`를 `[{wordId, activity}]` 형태로 확장, 하위호환은 기존 배열 포맷도 계속 읽도록 폴백) | "이번 주엔 특히 철자 쓰기에서 자주 틀렸어요: apple, banana, orange. 발음이나 뜻은 잘 알고 있어요." | 철자면 손으로 3번 써보기, 발음이면 같이 소리 내어 따라 하기 — **유형별로 다른 행동 제안** |

### 3.4 꾸준함 (Consistency) — 비난이 아니라 격려로 읽히는 프레이밍

같은 원시 데이터(연속학습일수, 최근 학습 빈도)를 쓰더라도 **문구 설계가
핵심**이다. 기존 `buildWeeklyReport()`도 이미 "숙제 안 한 날 N일"을
쓰지만, 이 문서는 프레이밍을 더 다듬는 제안이다.

| 항목 | 계산 근거 | 상태 | 문구 예시(before → after) | 학부모 다음 행동 |
|---|---|---|---|---|
| **결측일 표현 방식** | `student_daily_progress.date`(최근 7/14일 중 기록 없는 날 수) | 🟢 지금 바로 가능(기존 `buildWeeklyReport`의 `improve` 로직 문구만 개선) | before: "숙제를 안 한 날이 3일 있었어요." → after: "이번 주 4일은 꾸준히 했어요 — 나머지 3일도 짧게라도 이어가면 습관이 더 단단해질 거예요." (부족함을 먼저 말하지 않고 잘한 부분을 먼저 말한 뒤 격려로 연결) | 매일이 아니어도 "이번 주 5일 중 4일" 같은 현실적 목표로 같이 이야기하기 |
| **연속 기록이 끊긴 직후의 재시작** | `student_progress.streak`(현재값) + 과거 최대 streak(신규 컬럼/계산 필요 — 현재 `streak`는 "현재" 값만 저장, 과거 최고 기록 컬럼 없음) | 🟡 수집 추가 필요(`student_progress`에 `best_streak` 컬럼 추가 — upsert 시 `greatest(best_streak, streak)`로 갱신, 규칙 9/10 GRANT 동반) | "지금 연속 5일째예요. 예전에 8일까지 이어간 적도 있으니 이번에도 충분히 할 수 있어요!" ## 5. 구현 시 참고할 기존 규칙 (`CLAUDE.md`)

- 모든 신규 항목은 규칙 기반 계산만(AI API 호출 없음) — 규칙 7.
- 학생 식별은 항상 `students.id`(UUID) — 이 문서의 모든 계산은 이미
  `studentId` 기준으로 오는 `fetchDashboardData` 결과를 쓰므로 자연히
  준수됨 — 규칙 4.
- 새 컬럼(`first_activity_at`, `best_streak` 등) 추가 시 `students` 테이블이
  아니라 `student_daily_progress`/`student_progress`라 GRANT 요건이 다름 —
  이 두 테이블은 이미 "allow anon all" RLS 정책이라 컬럼 단위 GRANT는
  불필요(`DATABASE.md` "RLS/컬럼권한 현황" 절, `students`만 컬럼 단위
  GRANT 대상). 다만 여전히 멱등 마이그레이션(`add column if not exists`)
  원칙은 지켜야 함 — 규칙 9.
- `word_status` 이력화처럼 스키마 변경이 필요한 항목은 SQL 파일만 준비하고
  실행은 운영자 — 규칙 8.
- 이 문서 자체는 `handoff.md`/`ROADMAP.md`를 대체하지 않는 별도 조사
  산출물이다 — 실제 채택 여부는 운영자/구현 세션의 후속 판단.

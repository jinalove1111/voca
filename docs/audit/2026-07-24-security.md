**재현(라이브 실측, 실제 실행 결과)**:
```
PATCH .../rest/v1/classes?id=eq.<fake-uuid>  {"name":"..."}   → [] (권한거부 아님, 0행매칭만)
DELETE .../rest/v1/classes?id=eq.<fake-uuid>                   → [] (권한거부 아님, 0행매칭만)
POST   .../rest/v1/classes  {"name": 12345}                    → 200, 실제 행 생성됨 (즉시 정리함)
POST   .../rest/v1/units {}  → 23502(class_id not-null 위반, 42501 아님 = INSERT 권한 자체는 있음)
POST   .../rest/v1/words {}  → 23502(unit_id not-null 위반, 동일)
GET    .../rest/v1/classes?select=*&limit=1 → 실제 데이터 반환(실제 반 이름/설정 노출)
```

**코드상 근거**: `src/utils/wordLibrary.js:239,242,449-465,486,499,510,551,564,614,622,636` — 반/유닛/단어 생성·수정·삭제가 전부 anon key로 직접 Supabase에 쓴다. 이 흐름을 여는 유일한 게이트는 `AdminScreen.jsx`의 클라이언트 state(`authed`)뿐이며, 서버 측 재인증(`checkAdminReauth`, `api/_pinAuth.js:75`)은 이 경로 어디에도 없다 — PIN 관련 파괴적 액션 전부가 2026-07-16 P7 감사 후 이 패턴을 예외 없이 적용받은 것과 정확히 대비된다. `DATABASE.md:243`는 이 3테이블에 대해 "RLS/GRANT SQL이 저장소에 없다"를 미확인 기술부채로만 기록했을 뿐, 실측으로 "완전 개방"임을 확인하거나 Critical로 격상한 적은 없었다.

**왜 100명×20학원 규모에서 치명적인가**: anon key는 배포된 JS 번들에 그대로 포함되는 공개 값이다(PIN조차 필요 없음). 즉 로그인 여부와 무관하게 인터넷상 누구나 curl 몇 줄로 한 학원의 반/유닛/단어(정답 포함) 전체를 rename/삭제/오염시킬 수 있다. (a) 20개 학원이 각자 독립 배포(별도 Supabase 프로젝트)라면 이 취약점은 학원마다 독립적으로 Critical — 20곳 전부가 "관리자 PIN조차 필요 없는" 데이터 파괴 벡터를 각각 안고 있다는 뜻이라 총 기대 사고 건수가 20배. (b) 만약 향후 단일 Supabase 프로젝트로 20개 학원을 함께 서빙하는 구조로 통합한다면(`classes`에 `academy_id` 같은 경계 컬럼이 전혀 없음 — 실측 확인, 컬럼은 `id/name/class_type/created_at/spelling_test_enabled/spelling_hint_enabled/wrong_answer_repeat_count/spelling_direction`뿐), 단 하나의 anon key로 20개 학원 전체의 교육과정을 동시에 파괴할 수 있어 파국적.

**권장 수정**: `classes`/`units`/`words`에 `enable row level security` + anon/authenticated write(INSERT/UPDATE/DELETE) 거부 정책(SELECT는 기존처럼 열어둘지 학원 격리 설계와 함께 재검토) 적용(SQL 마이그레이션), 그리고 `wordLibrary.js`의 해당 쓰기 호출들을 `checkAdminReauth()`를 쓰는 신규 서버리스 엔드포인트(또는 `admin-pin-actions.js`류 dispatcher, Vercel Hobby 12개 함수 한도 고려)로 옮기는 코드 리팩터링이 함께 필요(코드 변경 + SQL 마이그레이션 둘 다 필수, SQL만으로는 기존 AdminScreen 플로우가 깨짐).

**지금 안전하게 고칠 수 있는가**: 아니오(단독 SQL만으로는) — 단, **2026-07-24 같은 날 별도 세션에서 이 문제를 코드(신규 Edge Function `admin-content-write` + `wordLibrary.js` 이관) + SQL(`supabase_v3_11_lockdown_curriculum_write.sql`) 짝으로 착수·구현 완료했다(`.ai-status/implementer-security-lockdown-curriculum.json` 참고, 배포는 운영자 수동 대기).**

### [High→Medium 재평가] 관리자 PIN 브루트포스 — 고정 지연만으로는 병렬 요청에 무력(기존 인지 항목의 정량 보강 + 신규 표면 추가)

**재확인**: `api/verify-admin-pin.js:25`, `api/_pinAuth.js:75-87`(`checkAdminReauth`) 모두 실패 시 고정 1.5초 지연만 있고 영속 카운터/잠금이 없음 — `handoff.md` 2026-07-16/2026-07-23에 이미 Medium으로 기록됨(재발견 아님).

**신규 관찰(정량화)**: Vercel 서버리스는 수평 확장되므로 이 1.5초는 "요청당" 지연일 뿐 "전체 처리량"을 제한하지 않는다. 동시 연결 수십~수백 개(스크립트 킷 수준으로 충분히 가능)면 4자리 숫자 PIN(10,000가지)을 수 분 내 전수조사 가능 — "온라인 브루트포스를 초당 수십 회로 늦춘다"는 코드 주석의 가정(순차 공격자)이 병렬 공격자에게는 성립하지 않는다.

**신규 표면**: 동일한 `ADMIN_PIN` 비밀값이 `supabase/functions/grade-writing-answers/index.ts:502-506`(별도 인프라, Supabase Edge Function)에서도 똑같은 "고정 1.5초 지연만" 패턴으로 재검증된다(`pipeline.js:545-548` `verifyAdminPin`). 즉 공격자는 같은 비밀번호를 공격할 수 있는 **독립된 두 표면**(Vercel + Supabase Edge)을 가지며, 이는 기존 감사에서 명시적으로 짚은 적 없는 조합.

**왜 규모에서 문제인가**: 이 저장소 현재 상태에서 실측한 `.env.local`의 `ADMIN_PIN`은 4자리 숫자가 아니라 영숫자 혼합 6자(코드가 요구하지 않음에도 운영자가 스스로 강화)였다 — 즉 *이번* 학원은 사실상 안전하나, 코드가 형식/강도를 전혀 강제하지 않으므로 20개 학원 중 한 곳이라도 운영자가 `0000`류 약한 값을 넣으면(학생 PIN에는 `isWeakPin()` 서버 거부가 있지만 관리자 PIN에는 형식 검증 자체가 없음, `api/verify-admin-pin.js:19`) 그 학원만 즉시 몇 분 내 뚫릴 수 있다. 20곳으로 늘어나면 "가장 약한 고리 1곳"이 나올 기대확률이 실질적으로 올라간다.

**권장 수정**: (1) 코드 차원 — 관리자 PIN 형식/최소 강도 서버측 검증 추가(약한 값 거부, 학생 PIN과 동일 원칙). (2) 진짜 rate limit/lockout은 서버리스 특성상 공유 상태 저장소(Upstash Redis/Vercel KV 등)가 필요 — 이는 `CLAUDE.md` 규칙 6(외부 의존성 최소화)과 정면으로 부딪히는 **정책적 트레이드오프**이며 운영자가 이미 여러 세션째 "과설계 안 함"으로 보류해온 결정(`handoff.md` 2026-07-16/23 기록). 코드만으로 해결 불가.

**지금 안전하게 고칠 수 있는가**: 형식 강제(1)는 예/작음(Low risk, 독립적 소패치, 아직 미구현). 진짜 rate limit(2)은 아니오 — 운영자 정책 결정 + 신규 의존성 도입이 선행돼야 함.

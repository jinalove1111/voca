## 추가 교훈 (`handoff.md`에서, 18개 규칙에 직접 포함되지 않은 것)

- **훅 순서 규칙이 조건부 return보다 우선한다.** `App.jsx`의
  `AppInner`는 `if (!studentData.restoreChecked) return <로딩화면>`을
  반드시 모든 훅 호출 뒤에 배치해야 함(React Hook 규칙 위반 방지) —
  새 조건부 early-return을 추가할 때 이 순서를 어기면 런타임 에러.
  근거: `DEVELOPER_GUIDE.md` Hook Rules.
- **취소 가능한 비동기 반복 작업은 반드시 cleanup에서 취소해야 한다.**
  React StrictMode의 개발 모드 effect 이중 실행이 실제로 "반복재생
  겹침(에코)" 프로덕션 버그를 낸 적이 있음 — 이후 `speech.js`의
  `playRepeating()`처럼 정리 함수를 반환하는 공용 함수로만 반복 작업을
  만드는 관례가 생김.
- **localStorage 쓰기 실패는 절대 throw하지 않고 삼켜야 한다.**
  `saveStore()`가 `setState` updater 안에서 호출되므로, 여기서
  throw하면 렌더 중 예외로 앱 전체가 크래시함(2026-07-16 P7 감사 수정).
- **fetch→setState 경로는 stale 응답 레이스에 취약하다.** 사용자가
  빠르게 다른 항목을 선택하면 먼저 보낸(느린) 요청의 응답이 나중에
  도착해 최신 선택을 덮어쓸 수 있음 — PIN 만들기 탭에서 실제
  프로덕션 버그로 발견(`6dd6c7a`), 이후 유사 패턴 4곳을 전수 점검해
  가드 추가(`529ff9e`). 상세는 [`wiki/bug-history.md`](./bug-history.md).
- **번들 해시 대조로 실제 배포를 확인하는 습관.** "push는 했지만
  Vercel 배포가 아직 안 끝났거나 실패한" 상황을 로그가 아니라 실제
  서빙되는 번들 파일명 + 그 안의 고유 문자열로 확정하는 반복 습관 —
  `git push` 자체를 배포 완료의 증거로 삼지 않음. 근거:
  `ARCHITECTURE.md` "8. 배포 프로세스".
- **테스트는 로직을 손으로 베껴 재구현하지 않는다.** React 훅/컴포넌트
  테스트도 esbuild로 실제 소스를 번들해서 실행 — "테스트가 통과하는데
  실제 버그가 남아있는" 괴리를 방지. 근거: `TESTING.md` "핵심 원칙".

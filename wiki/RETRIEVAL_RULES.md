# wiki/RETRIEVAL_RULES.md — 검색·검증 규칙 (향후 에이전트용)

_이 문서는 이 위키를 읽는(또는 `wiki:search`로 검색하는) 모든 향후
세션이 지켜야 하는 규칙입니다. `CLAUDE.md` 저장소 헌법과 함께 최상위
우선순위로 취급하세요._

## 핵심 규칙: 위키는 코드보다 우선하지 않는다

**이 위키의 어떤 사실도 코드 자체보다 우선하지 않습니다.** 위키
페이지에서 `파일:줄번호` 형태의 참조를 발견하면:

1. 그 참조가 가리키는 파일을 **직접 열어** 지금도 그 내용이 맞는지
   확인합니다.
2. 코드/스키마가 위키 작성 시점(각 페이지 상단 `_작성: ...`)보다 나중에
   바뀌었을 수 있습니다 — 확인 없이 위키 문구를 그대로 신뢰해 코드를
   작성/판단하지 않습니다.
3. 위키와 원본 문서(`PROJECT_GUIDE.md` 등 6종 + `handoff.md`)가
   어긋나면, **원본 문서가 우선**합니다. 위키는 그 원본의 요약일 뿐
   별도의 진실 원천이 아닙니다.
4. 위키와 코드가 어긋나면, **코드가 우선**합니다(원본 문서도 마찬가지 —
   `DEVELOPER_GUIDE.md`의 AI 세션 표준 워크플로우 4번 단계와 동일 원칙).

## `CLAUDE.md` 규칙 3(재구현 금지)과의 연계

`CLAUDE.md` 규칙 3은 "완료로 선언된 작업은 재구현하지 않는다"고
명시합니다 — 이 저장소는 재조사 없는 재작업이 이미 검증된 로직을 조용히
되돌려 회귀를 만든 실제 사고 이력이 있습니다(`unit_name` 문자열 매칭
방식이 v2.1에서 이미 `current_unit_id`로 교체됐는데, 배경을 모르고 다시
문자열 매칭으로 "새로" 구현하면 고친 버그가 재발하는 사례).

이 위키(특히 [`wiki/decisions.md`](./decisions.md),
[`wiki/bug-history.md`](./bug-history.md))는 정확히 이 규칙을 지키기
위한 **빠른 조회용 색인**입니다 — "이거 이미 했던 결정 아닌가?"를
`handoff.md` 1800줄+ 전체를 읽지 않고 먼저 훑어볼 수 있게 하는 목적.
단, 위키에서 "이미 완료됨"을 봤다고 그대로 믿지 말고, 반드시:

1. 위키가 링크한 원본 문서(`ROADMAP.md`/`handoff.md`의 해당 섹션) 또는
   커밋을 직접 열어 확인
2. 관련 코드가 지금도 그 상태인지 실제로 grep/read

두 단계를 거친 뒤에만 "완료됐다"고 판단하세요.

## 검색 명령 사용 규칙

`npm run wiki:search -- "키워드"`(`scripts/wikiSearch.mjs`)는 키워드가
포함된 **라인 발췌**를 관련도 순으로 보여줄 뿐, 그 발췌가 여전히
유효한지는 검증하지 않습니다. 검색 결과에서 유용해 보이는 참조를 찾으면:

1. 출력된 `파일:줄번호`를 그대로 `Read` 도구로 열어 전후 맥락까지 확인
2. 그 파일이 `handoff.md`처럼 세션 로그면, 더 최근 세션에서 같은 주제를
   덮어쓰거나 갱신한 게 없는지 위에서(최신이 상단) 훑어보기
3. 코드 파일이면 현재 브랜치 최신 상태인지(`git log -1 -- <파일>`) 확인

## 위키 자체 갱신 규칙

- 이 위키는 기존 문서의 append-only 원칙(`CLAUDE.md` 규칙 13)을
  그대로 따릅니다 — 각 페이지에 새 사실을 추가할 때는 기존 섹션을
  삭제/재작성하지 않고 이어서 추가합니다.
- 새 사실은 반드시 원본 문서/코드/커밋에서 검증한 뒤에만 추가합니다 —
  추측이나 "아마 이럴 것"은 기록하지 않습니다(불확실하면 "확인 필요"로
  정직하게 표기 — `wiki/api-costs.md`의 Supabase 요금제 항목이 실례).
- 새 하위 페이지가 필요해지면 이 `RETRIEVAL_RULES.md`와
  [`wiki/HOME.md`](./HOME.md)의 표에 함께 추가합니다.

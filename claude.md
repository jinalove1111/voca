# Paul Easy Voca — 저장소 헌법 (Repository Constitution)

_이 섹션이 이 파일의 최상위 우선순위입니다. 아래 18개 규칙은 이 저장소에서
작업하는 모든 세션(사람/AI 에이전트 무관)이 다른 어떤 지시보다 우선 따라야
합니다. 프로젝트 배경/로드맵 등 나머지 정보는 이 섹션 아래, 또는
`PROJECT_GUIDE.md` 등 별도 문서로 이동했습니다 — "무엇을 하는 앱인지"가
궁금하면 아래 참조 섹션을, "어떻게 작업해야 하는지"가 궁금하면 지금 이
섹션을 보세요._

## 필수 선행 조사

새 세션은 코드를 건드리기 전에 최소한 `PROJECT_GUIDE.md`(진입점 + 헷갈리는
것 Top 5)를 읽습니다. 작업 범위에 따라 `ARCHITECTURE.md`/`DATABASE.md`/
`DEVELOPER_GUIDE.md`/`TESTING.md`/`ROADMAP.md`도 함께 — 상세 매핑은
`DEVELOPER_GUIDE.md`의 "AI 세션 표준 워크플로우" 참고.

## 18개 규칙

1. **안정성 최우선 — 기존 기능을 절대 깨뜨리지 않는다.** 새 코드가 기존
   플로우(로그인/학습/퀴즈/동기화 등)를 조금이라도 위험하게 하면 범위를
   줄이거나 중단한다.
2. **새 기능보다 기존 버그 수정이 항상 우선한다.**
3. **완료로 선언된 작업은 재구현하지 않는다.** 이 저장소는 재조사 없는
   재작업이 이미 검증된 로직을 조용히 되돌려 회귀를 만든 실사고 이력이
   있다 — 예: `unit_name`(문자열) 매칭 기반 유닛 판정이 표기 차이("Unit 1"
   vs "Unit1")에 취약했던 구조를 v2.1이 `current_unit_id`(FK)로 이미
   교체했는데(`handoff.md` "학생-Unit 아키텍처" 섹션), 이 배경을 모르고
   문자열 매칭 방식으로 "새로" 구현하면 이미 고친 버그가 재발한다. 기존
   구현이 왜 지금 형태인지 `handoff.md`/`ROADMAP.md`에서 먼저 확인한다.
4. **사용자(학생) 식별은 항상 `students.id`(UUID)로 하고, 이름 문자열로
   매칭하지 않는다.** v1.6 이전에는 이름이 사실상 전역 유일 키였고, 동명이인
   학생이 서로의 별/포인트/캘린더/학습기록을 덮어쓰는 실제 프로덕션
   데이터무결성 사고가 있었다(CTO 최우선순위 P0 대응 이력, `handoff.md`
   "v1.6 — 학생 identity P0 리팩터링" 섹션). 새 코드/쿼리/캐시 키 어디에도
   학생 이름을 유일 식별자로 쓰지 않는다.
5. **매 작업마다 build → 오류 확인 → 관련 verify 하네스 실행.** 최소
   `npm run build` 통과, 건드린 도메인의 `npm run verify:<domain>`(또는
   최종적으로 `npm run verify:all`)까지 확인 후 마감한다.
6. **외부 의존성 최소화.** 이미 있는 라이브러리/Node 내장으로 해결 가능하면
   새 패키지를 추가하지 않는다(PIN 해싱을 `bcrypt` 대신 Node 내장
   `crypto`(scrypt)로 구현한 것이 이 원칙의 실례, `api/_pinAuth.js`).
7. **비용이 드는 AI/유료 API 기능은 무료 대안을 먼저 찾는다.** 학부모 주간
   리포트가 실제 AI 호출 없이 규칙 기반 템플릿으로 구현된 것이 실례.
8. **에이전트/CI는 Supabase에 DDL을 직접 실행할 수 없다.** 새 테이블/컬럼이
   필요하면 `supabase_v{n}_{설명}.sql` 파일만 준비하고, 실행은 운영자가
   Supabase 대시보드 SQL Editor에서 수동으로 한다.
9. **모든 DB 마이그레이션은 멱등이어야 하고, 코드보다 먼저 실행되든 나중에
   실행되든 앱이 깨지지 않아야 한다.** 클라이언트는 새 컬럼/테이블 부재를
   감지해 안전한 기본값으로 폴백해야 한다.
10. **`students` 테이블에 새 컬럼을 추가하면 GRANT를 반드시 함께 실행한다.**
    v1.9로 테이블 단위 SELECT/UPDATE가 회수된 상태라, 새 컬럼의
    `grant select/update`를 빠뜨리면 그 컬럼뿐 아니라 기존에 잘 동작하던
    조회까지 fail-closed로 깨질 수 있다(`DATABASE.md` RLS/컬럼권한 섹션).
11. **PIN/자격증명 관련 컬럼(`pin_hash`/`pin_fail_count`/`pin_locked_until`/
    `pin_setup_allowed`)은 클라이언트 코드가 절대 조회/로깅하지 않는다.**
    PIN 검증/설정은 오직 `api/*.js`(서버리스, service_role key)만 한다.
12. **학생 대상 신규 기능/UI/게임화는 이번 "AI 개발 운영체제" 구축 범위에서
    절대 금지.** 이 문서와 `.claude/agents/`, `.ai-status/`, 훅, 프로젝트
    보드는 순수 개발 인프라이며 학생이 보는 화면/로직과 무관하다.
13. **문서 갱신은 append-only가 원칙이다.** `handoff.md`/`ROADMAP.md`/
    `DEVELOPER_GUIDE.md`/`TESTING.md`/`DATABASE.md`/`PROJECT_GUIDE.md`의
    기존 섹션은 삭제·재작성하지 않고 새 섹션을 위(또는 규칙에 맞는 위치)에
    추가한다. 이 `CLAUDE.md`의 "저장소 헌법" 섹션은 예외(의도적 재구성).
14. **작업 단위로 나눠 소커밋한다.** 큰 변경(리팩터링 등)도 파일/기능
    단위로 쪼개 커밋하고, 한 커밋에 여러 무관한 변경을 섞지 않는다.
15. **회귀가 의심되면 수정 전 코드로 되돌려 테스트가 실제로 FAIL하는지
    먼저 확인한다.** "테스트 자체의 유효성 검증" 패턴 — `syncGenRef`
    세대 카운터 수정 시 실제로 이 방식으로 회귀를 재현·확정했다
    (`handoff.md` 2026-07-18 Phase 1 섹션).
16. **파일당 실제 구현 소유자는 한 세션(1명/1에이전트)만이다 — 동시에 다른
    에이전트가 작업 중인 파일은 읽기만 하고 쓰지 않는다.** 이 저장소에서
    실제로 서로 다른 두 에이전트의 변경이 한 커밋(`55f0c86`)에 섞여 들어간
    사고가 있었다 — 한 에이전트가 `handoff.md`를 `git add`한 뒤 커밋하는
    과정에, 동시에 작업 중이던 다른 에이전트가 이미 스테이징해 둔 데드코드
    삭제(4개 파일)가 의도치 않게 같은 커밋에 함께 실렸다(기능적으로는
    무해했지만 커밋 이력의 attribution이 부정확해졌다, `handoff.md`
    2026-07-18 Phase 4 섹션 상단 참고). 작업 시작 전 `git log`/`git status`로
    동시 작업 흔적을 확인하고, 자신이 손대지 않은 파일은 절대 `git add`
    범위에 넣지 않는다.
17. **의미있는 체크포인트마다 `.ai-status/`에 상태 파일을 기록한다.** 이는
    Claude 내부 상태를 노출하는 게 아니라 순수 파일 기반 관례다 — 스키마는
    `.ai-status/README.md` 참고.
18. **훅으로 실제 강제되는 것과 문서로만 강제되는 것을 구분해 정직하게
    기록한다.** 예: SQL 파일의 파괴적 패턴(DROP/TRUNCATE/무조건부 DELETE 등)
    감지·차단은 `.claude/settings.json`의 PreToolUse 훅으로 실제 강제된다.
    반면 "완료 선언 시 자동으로 verify를 실행시키는 것"처럼 텍스트 의미를
    해석해야 하는 강제는 이 환경에서 신뢰성 있게 불가능하므로, 훅이 아니라
    `DEVELOPER_GUIDE.md`의 프로세스 규칙(사람/에이전트의 자율 준수)으로만
    다룬다 — 안 되는 걸 되는 것처럼 문서화하지 않는다.

## 필수 완료 체크리스트

작업을 "완료"로 보고하기 전에 전부 확인:

- [ ] `npm run build` 통과 (에러/신규 경고 없음)
- [ ] 건드린 도메인의 `npm run verify:<domain>` (또는 `verify:all`) 실행·확인
- [ ] 관련 문서(`handoff.md` 최소, 필요 시 `ARCHITECTURE.md`/`DATABASE.md`/
      `DEVELOPER_GUIDE.md`/`TESTING.md`/`ROADMAP.md`)에 append
- [ ] 자신이 소유하지 않은 파일을 커밋에 포함하지 않았는지 `git status`/
      `git diff --staged`로 확인 (규칙 16)
- [ ] 학생 식별에 이름이 아니라 UUID를 썼는지 확인 (규칙 4)
- [ ] 새 Supabase 컬럼/테이블에 GRANT·폴백이 필요하면 빠짐없이 처리했는지
      확인 (규칙 9, 10)
- [ ] 학생 대상 기능/UI/게임화를 이번 작업 범위에 섞지 않았는지 확인
      (규칙 12)
- [ ] `.ai-status/`에 체크포인트 상태 파일을 기록했는지 확인 (규칙 17)
- [ ] 파일/기능 단위 소커밋으로 나눴는지 확인 (규칙 14)

# 프로젝트 배경 (참조용 — 최신 상태는 아래 문서가 우선)

_이 아래는 최초 기획서(MVP 1단계 시점) 원본을 요약한 배경 정보입니다.
**"지금 무엇이 구현돼 있고 무엇이 남았는지"는 이 섹션이 아니라
`PROJECT_GUIDE.md`(진입점)와 `ROADMAP.md`(버전별 완료 현황)를 봐야 합니다**
— 이 섹션은 프로젝트 목적/사용자 역할처럼 아직 유효한 배경만 남기고,
MVP 1단계 시점의 상세 설계(반 목록 하드코딩, DB 스키마 초안 등)는 이미
`DATABASE.md`/`ARCHITECTURE.md`로 대체되어 생략했습니다._

## 프로젝트 목적

폴이지보카(Paul Easy Voca)는 영어 공부방 학생들이 매일 단어를 학습하고,
발음 듣기·녹음 후 따라 말하기로 발음 실력을 향상시키며, 학습 기록과 숙제
완료 상태를 관리하는 앱입니다. 현재 111명 규모 학생이 실사용 중이며
production-ready 판정을 받았습니다(`ROADMAP.md` 2026-07-18 Production
Readiness 감사 섹션).

## 사용자 역할

- **학생**: 이름+PIN 로그인, 반/유닛별 단어 학습, 발음 듣기·녹음, 퀴즈/쓰기
  시험, 숙제 완료 상태 확인.
- **관리자(원장)**: 반/단어 업로드·수정, 학생-반 매핑 관리, 숙제·학습 기록
  확인, 반/날짜별 학습 스케줄 관리.
- **학부모**: 학생 학습 진도·숙제 완료 여부·발음/퀴즈 기록 조회(읽기 전용).

## 문서 지도

`PROJECT_GUIDE.md`의 "문서 지도" 표가 최신 기준입니다. 요약:

| 문서 | 용도 |
|---|---|
| `PROJECT_GUIDE.md` | 진입점, 빠른 시작, 헷갈리는 것 Top 5 |
| `ARCHITECTURE.md` | 전체 구조/인증/상태관리/캐싱/영속성/배포/주요 플로우 |
| `DATABASE.md` | 테이블/FK/마이그레이션 순서/RLS |
| `DEVELOPER_GUIDE.md` | 코딩 규칙 + 체크리스트 + AI 세션 표준 워크플로우 |
| `TESTING.md` | 테스트 체계 + verify 하네스 |
| `ROADMAP.md` | 버전별(v1.0~) 완료 현황과 백로그 |
| `handoff.md` | 세션별 상세 작업 로그(최상단이 최신) — 사실상의 진실 원천 |
| `PROJECT_BOARD.md` | 작업 보드(BACKLOG/NEXT/IN_PROGRESS/VERIFY/DONE/BLOCKED) |
| `.claude/agents/*.md` | 역할별 에이전트 정의(planner/implementer/qa-reviewer/security-reviewer/docs-maintainer) |
| `.ai-status/` | 파일 기반 에이전트 상태 프로토콜(README.md에 스키마) |

`claude.md`(소문자, 이 파일과 Windows에서 동일 파일)/구 기획서 스타일의
나머지 세부 항목(반 이름 하드코딩, DB 컬럼 초안 등)은 전부 `DATABASE.md`/
`ARCHITECTURE.md`의 실제 구현 기준 문서로 대체되었으므로 여기서는 생략합니다.
